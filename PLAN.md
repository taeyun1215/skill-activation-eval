# Skill Activation Eval — 전체 실험 계획

## 연구 질문

"CLAUDE.md에 어떤 규칙을 써야, 에이전트가 스킬대로 동작하는가?"

Claude Code에서 커스텀 에이전트(`.claude/agents/`)와 스킬(`.claude/skills/`)을 정의해도, 모델이 이를 올바르게 활용하는지는 보장되지 않는다. 어떤 설정 조합이 가장 높은 정확도를 만드는지 실험적으로 확인한다.

## 실험 순서

각 실험은 이전 실험의 결과 위에 쌓인다. 순서대로 진행.

```
01 훅이 스킬 호출을 유도하는가?           ✅ 완료
   ↓ (결과: forced-eval 훅 = 100%)
02 위임 지시 + 에이전트 설명이 영향을 주는가?
   ├─ v1 위임 지시 강도                   ✅ 완료 (강도 무관)
   ├─ v2 에이전트 설명 수준               ✅ 완료 (detailed 62%)
   ├─ v3 디스크립션 교차참조 적용          ✅ 완료 (name-only 75%)
   ├─ v4 훅 기반 에이전트 선택 가이드      ✅ 완료 (forced-eval 82%, N=40)
   └─ v5 Step 0 요청 분류 추가            ✅ 완료 (87%, N=40)
   ↓ (전제: 올바른 서브에이전트 선택이 안정화된 후)
03 스킬 전달 방식이 영향을 주는가?         ← 다음
```

## 01 — 훅 기반 스킬 활성화

### 배경

Claude Code에는 `UserPromptSubmit` 훅이 있다. 사용자 프롬프트가 제출될 때 셸 스크립트를 실행하여 시스템 메시지를 주입할 수 있다. 이걸로 "Skill() 도구를 사용하세요"라고 강제하면 스킬 호출률이 올라갈까?

### 설정

| 항목 | 내용 |
|------|------|
| 질문 | UserPromptSubmit 훅으로 Skill() 호출을 유도할 수 있는가? |
| 독립변수 | 훅 종류 (none, simple, forced-eval, llm-eval) |
| 측정 | Skill() 호출 여부 |
| 디렉토리 | `01-hook-activation/` |

### 결과

| Config | 정확도 |
|--------|--------|
| none | 35% |
| simple | 60% |
| llm-eval | 95% |
| forced-eval | **100%** |

### 결론

훅으로 시스템 수준 주입하면 100% 달성. **단, 서브에이전트에는 훅이 전파되지 않는다.** 메인 에이전트에서만 동작.

### → 02로의 전환

01에서 "메인 에이전트의 행동은 훅으로 제어 가능"을 확인했다. 그런데 실제 프로젝트에서는 메인 에이전트가 직접 코드를 쓰지 않고, **서브에이전트에 위임**한다. 위임할 때 "어떤 서브에이전트를 선택하느냐"가 다음 문제다. CLAUDE.md에 어떤 지시를 써야 올바른 에이전트를 고르는지 실험한다.

## 02 — 서브에이전트 위임 및 선택

### v1 — 위임 지시 강도

#### 배경

가장 단순한 가설: "CLAUDE.md에 '반드시 위임하세요'라고 강하게 쓰면 위임률과 정확도가 올라갈 것이다." 지시 강도를 4단계로 나누어 테스트.

#### 설정

| 항목 | 내용 |
|------|------|
| 질문 | CLAUDE.md 위임 지시 강도가 에이전트 선택 정확도에 영향을 주는가? |
| 독립변수 | Agent Delegation 지시 강도 (none → mention → should → must) |
| 환경 | `.claude/agents/` 6개 제공, **에이전트 이름만 나열** (설명 없음) |
| 테스트 | 16건 × 4 configs × 1R |

#### 결과

| Config | 위임률 | 정확도 |
|--------|--------|--------|
| none | 75% | 31% |
| mention | 87% | 50% |
| should | 68% | 50% |
| must | 81% | 50% |

#### 결론

**위임 지시 강도는 의미 없었다.** none도 75% 위임하고, must도 81%. 차이 미미. 정확도는 전부 50% 부근 — 때려맞추기 수준.

문제는 "위임을 안 한다"가 아니라 **"올바른 에이전트를 못 고른다"**였다. `db-query-helper`, `migration-planner` 같은 낯선 이름의 에이전트는 선택률 0%. 모델이 에이전트가 뭘 하는지 모르니까 익숙한 것(code-analyzer, Explore)만 고른다.

#### → v2로의 전환

v1에서 병목이 "에이전트 설명 부재"임을 확인. 그러면 CLAUDE.md에 에이전트 설명을 얼마나 자세히 써야 정확도가 올라가는지 테스트한다.

---

### v2 — 에이전트 설명 수준

#### 배경

v1에서 이름만으로는 50%가 한계. CLAUDE.md의 `Available Agents` 섹션에 설명을 추가하면 될까? 설명 수준을 4단계로 나눈다.

#### 설정

| 항목 | 내용 |
|------|------|
| 질문 | CLAUDE.md에 에이전트 설명을 어떻게 제공하면 정확도가 올라가는가? |
| 독립변수 | 설명 수준 (name-only → one-liner → detailed → read-file) |
| 고정 | Agent Delegation = must |

| Config | Available Agents 섹션 내용 |
|--------|--------------------------|
| name-only | 이름만 나열 (v1 must와 동일 베이스라인) |
| one-liner | 이름 + 한줄 설명 (frontmatter description) |
| detailed | 이름 + 2~3줄 + 적합/비적합 태그 |
| read-file | 이름 + "`.claude/agents/{이름}.md`를 Read하세요" |

#### 결과

| Config | 위임률 | 정확도 |
|--------|--------|--------|
| name-only | 75% | 56% |
| one-liner | 87% | 50% |
| detailed | 68% | **62%** |
| read-file | 68% | 56% |

#### 핵심 발견

- **detailed가 62%로 최고** — 적합/비적합 태그가 에이전트 구분에 도움
- **v1에서 0%였던 migration-planner가 4/4 성공** — 설명만 추가하면 미지 에이전트도 선택 가능
- **db-query-helper 첫 성공** — detailed에서 N03 케이스 1건 성공. v1에서 완전 0%였던 에이전트
- **D03 "쿼리 최적화" 전멸** — "최적화" 키워드가 performance-optimizer로 끌림. 설명으로 해결 불가
- **위임률 역설**: 설명이 많을수록 위임률 하락 (68%). 에이전트 역할을 알게 되니 "이건 내가 직접 하겠다"고 판단

#### → v3로의 전환

v2 결과를 보면서 흥미로운 점을 발견했다. CLAUDE.md에 설명을 아무리 써도 62%가 한계인데, Claude Code가 `Task(subagent_type=...)` 도구를 생성할 때 내부적으로 **에이전트 .md 파일의 `description` 프론트매터**를 참조한다는 것을 알게 되었다. 그러면 CLAUDE.md가 아니라 에이전트 .md 파일 자체를 개선하면 어떨까? 특히 에이전트 간 혼동을 방지하는 **교차참조**("쿼리 성능 문제는 performance-optimizer가 아니라 db-query-helper를 사용하세요")를 description에 추가해본다.

### v3 — 디스크립션 교차참조 적용

#### 배경

v2에서 발견한 핵심 인사이트: Claude Code는 에이전트 선택 시 CLAUDE.md보다 **에이전트 .md의 description 프론트매터**를 더 중요하게 본다. 이 description에 교차참조를 추가하면 에이전트 간 혼동이 줄어들까?

예: `db-query-helper.md`의 description에 "쿼리 성능이 느린 경우에도 이 에이전트를 사용하세요. performance-optimizer는 서비스 레벨 성능 최적화용입니다."

#### 설정

| 항목 | 내용 |
|------|------|
| 질문 | 에이전트 .md의 description에 교차참조를 추가하면 정확도가 올라가는가? |
| 변경 | 에이전트 description에 "X는 Y를 사용하세요" 교차참조 추가 |
| 고정 | v2와 동일 4 configs, must 위임 |

#### 결과 (vs v2)

| Config | v2 | v3 | 변화 |
|--------|-----|-----|------|
| name-only | 56% | **75%** | **+19%** |
| one-liner | 50% | 56% | +6% |
| detailed | 62% | 68% | +6% |
| read-file | 56% | 68% | +12% |

#### 핵심 발견

- **name-only가 75%로 1등** — 반전. CLAUDE.md에 아무 설명 없어도 description만 좋으면 75%
- **교차참조가 전 config에 영향** — description은 CLAUDE.md 설명과 무관하게 작용. Claude Code 내부 메커니즘
- **v2에서 전멸이던 D03 "쿼리 최적화"** → name-only/read-file에서 성공
- **N03 "쿼리가 느려"** → detailed에서 첫 성공. v2에서는 전멸

결론: **에이전트 .md의 description 프론트매터가 선택의 핵심 변수.** CLAUDE.md 설명 수준은 부차적.

#### → v4로의 전환

v3까지 CLAUDE.md와 에이전트 description을 최적화하여 75%에 도달했다. 그런데 01 실험에서 "훅으로 시스템 메시지를 주입하면 행동을 강제할 수 있다"는 것을 이미 증명했다. 01은 Skill() 호출이었지만, **같은 원리를 에이전트 선택에도 적용**하면 어떨까?

01의 `forced-eval` 훅은 "각 스킬을 1~5점으로 채점한 뒤 가장 높은 점수의 스킬을 호출하라"는 구조였다. 이걸 에이전트 선택에 맞게 변형하면: "각 에이전트의 적합도를 1~5점으로 채점하고, 가장 적합한 에이전트에 위임하라."

추가로, 다른 접근도 테스트한다. 외부 LLM(Haiku 4.5)에게 "이 프롬프트에 어떤 에이전트가 적합한가?"를 물어보고 그 추천을 주입하는 방식.

---

### v4 — 훅 기반 에이전트 선택 가이드

#### 배경

v3의 description 최적화로 75%. 01의 훅 강제로 스킬 100%. 두 접근을 결합한다: **description 교차참조(v3) + 훅 기반 강제 평가(01)**. 여기에 테스트 케이스도 16건 → 20건으로 확장하고, 신뢰도를 위해 2R 반복.

#### 설정

| 항목 | 내용 |
|------|------|
| 질문 | UserPromptSubmit 훅으로 에이전트 선택 가이드를 주입하면 정확도가 올라가는가? |
| 독립변수 | 훅 종류 (none, simple, forced-eval, llm-eval-haiku) |
| 고정 | CLAUDE.md = name-only + must (v3 베이스라인), 에이전트 description = v3 교차참조 |
| 테스트 | 20건 × 4 configs × 2R |

| Config | 훅 내용 |
|--------|---------|
| none | 훅 없음 (v3 name-only 베이스라인) |
| simple | "적합한 에이전트를 선택하세요" 단순 안내 |
| forced-eval | 에이전트 목록 + description 주입 + 강제 평가 지시 (Step 1→2→3) |
| llm-eval-haiku | Haiku 4.5가 프롬프트 분석 후 에이전트 추천 → Claude에 주입 |

#### 결과 (R1+R2 평균, 각 20건 × 2회 = 40건)

| Config | R1 | R2 | 평균 |
|--------|-----|-----|------|
| none | 55% | 65% | **60%** |
| simple | 70% | 70% | **70%** |
| forced-eval | 85% | 80% | **82%** |
| llm-eval-haiku | 70% | 70% | **70%** |

#### 유형별 정확도 (R1+R2 합산)

| Config | direct | natural | ambiguous |
|--------|--------|---------|-----------|
| none | 78% | 55% | 37% |
| simple | 92% | 61% | 50% |
| forced-eval | **100%** | **83%** | 50% |
| llm-eval-haiku | 92% | 55% | **62%** |

#### R1 vs R2 안정성

| Config | 둘다성공 | 둘다실패 | 불안정 |
|--------|---------|---------|--------|
| none | 11 | 7 | 2 |
| simple | 13 | 5 | 2 |
| forced-eval | 16 | 3 | 1 |

#### 핵심 발견

- **forced-eval 82%로 최고** — none 대비 +22%p, simple 대비 +12%p
- **"사고 과정 강제" > "정답 알려주기"** — 가장 놀라운 발견. Haiku가 정확한 에이전트를 추천해도 Claude가 무시하고 Explore로 보냄. llm-eval-haiku(70%) = simple(70%)과 동일. 모델에게 "답을 알려주는 것"보다 "스스로 생각하게 강제하는 것"이 효과적
- **llm-eval-haiku은 ambiguous에서만 강점** — 62%로 forced-eval(50%)보다 높음. 모호한 요청에 외부 LLM의 추가 판단이 도움
- **direct 100%** — 명시적 요청은 완벽 해결
- **natural이 핵심 차별점** — none 55% → forced-eval 83% (+28%p). 자연어 표현 이해에 구조화된 평가가 크게 기여
- **ambiguous 50% 천장** — forced-eval도 simple과 동일. 모호한 요청은 훅으로도 한계
- **N03 "쿼리가 느려" 전멸 지속** — v1부터 v4까지 "느려" 키워드가 perf-opt로 끌리는 편향 해결 불가
- **A03 "새 기능 추가" 전멸 지속** — 범위가 너무 넓어 위임 자체를 회피

#### → v5로의 전환

v4에서 82%를 달성했지만, 분석하면서 "왜 N03이 계속 실패하는가?"를 깊이 파고들었다. N03 "UserRepository의 findByNickname 쿼리가 너무 느려"는 expected가 db-query-helper인데, "느려"라는 키워드가 performance-optimizer를 강하게 연상시킨다. forced-eval에서 Step 1~3으로 각 에이전트를 평가하라고 지시해도, "느려 → 성능 → performance-optimizer"라는 **키워드 편향**을 극복하지 못한다.

여러 개선 방안을 검토했다:
1. 트리거/안티트리거 키워드 — 에이전트별로 추가. 비범용적이라 패스
2. 양방향 교차참조 — description 강화. 이미 v3에서 함
3. **Step 0 요청 분류** — 에이전트를 평가하기 전에 요청 자체를 먼저 분류
4. 폴백 규칙 — 실패 시 대안. 보조적이라 후순위

**Step 0의 핵심 아이디어**: 모델이 "느려"라는 키워드에 바로 반응하지 않도록, 먼저 "이 요청이 query-level인지 system-level인지" 구조적으로 분류하게 한다. N03은 "findByNickname 쿼리"라고 명시하므로 query-level로 분류되고, 그러면 db-query-helper가 자연스럽게 매칭된다. "분류 먼저, 평가 나중" 패턴.

---

### v5 — Step 0 요청 분류 추가

#### 배경

v4 forced-eval의 Step 1~3(에이전트별 적합도 채점) 앞에 Step 0을 추가한다. 에이전트를 보기 전에 요청의 구조적 속성을 먼저 파악하여 키워드 편향을 차단.

#### Step 0 내용

```
Step 0 — 요청 분류:
- 작업 레벨: query-level | service-level | system-level | design-level
- 작업 유형: 분석 | 작성/수정 | 테스트 | 설계 | 계획
- 대상 특정성: specific | general | vague
```

#### 설정

| 항목 | 내용 |
|------|------|
| 질문 | 에이전트 평가 전에 요청을 먼저 분류하면 키워드 편향을 극복할 수 있는가? |
| 변경 | v4 forced-eval의 Step 1~3 앞에 Step 0 추가 |
| 고정 | CLAUDE.md = name-only + must, description = v3 교차참조 |
| 테스트 | 20건 × 1 config × 2R |

#### 결과

| Config | R1 | R2 | 평균 |
|--------|-----|-----|------|
| v5 forced-eval | 85% | 90% | **87%** |
| (참고) v4 forced-eval | 85% | 80% | 82% |

#### 유형별 비교

| Config | direct | natural | ambiguous |
|--------|--------|---------|-----------|
| v5 | 100% | **94%** | 50% |
| v4 | 100% | 83% | 50% |

#### 개선된 케이스

| ID | Prompt | v4 | v5 | Step 0이 해결한 것 |
|----|--------|----|----|-------------------|
| N03 | "findByNickname 쿼리가 느려" | ✗/✗ | **✓/✓** | "query-level, specific" 분류 → 키워드 편향 극복 |
| N04 | "엣지 케이스가 걱정돼" | ✓/✗ | **✓/✓** | "테스트" 작업 유형 분류 → 안정화 |

## 03 — 스킬 전달 방식

### 배경

02에서 "올바른 서브에이전트를 선택하는 방법"을 87%까지 끌어올렸다. 다음 문제: 서브에이전트가 선택되었을 때, **해당 에이전트가 프로젝트 스킬을 따르는가?** Task 프롬프트에 스킬 정보를 어떻게 넘겨야 서브에이전트가 스킬대로 동작하는지 테스트한다.

### 설정

| 항목 | 내용 |
|------|------|
| 질문 | Task 위임 시, 스킬 정보를 어떻게 넘겨야 서브에이전트가 스킬대로 동작하는가? |
| 독립변수 | 스킬 전달 방식 (none → tag → path → embed) |
| 측정 | skill_loaded, skill_followed |
| 전제 | 02에서 올바른 서브에이전트 선택이 안정화된 후 |
| 디렉토리 | `03-skill-delivery/` |
| 상태 | ⏳ 다음 |

**Config 후보:**

| Config | Task 프롬프트에 스킬을 넘기는 방식 |
|--------|----------------------------------|
| none | `Task(prompt="쿼리 최적화해줘")` — 스킬 정보 없음 |
| tag | `Task(prompt="<!-- skill: entity-skill --> 쿼리 최적화해줘")` |
| path | `Task(prompt="`.claude/skills/entity-skill/SKILL.md`를 참고해서 쿼리 최적화해줘")` |
| embed | `Task(prompt="<skill-guide>{SKILL.md 내용}</skill-guide> 쿼리 최적화해줘")` |

> v1 시뮬레이션 결과: none 37%, tag 87%, light-embed 96%, direct-embed **100%**
> → E2E 검증 필요

---

## 핵심 발견

### 01 — 훅
1. **훅은 메인 세션에서만 동작** — 서브에이전트에는 훅이 전파되지 않음
2. **정보를 확정적으로 전달해야 100%** — 모델의 자발적 행동에 의존하면 최대 96%

### 02 — CLAUDE.md & Description
3. **위임 지시 강도는 의미 없음** — none도 75% 위임, must도 81%. 차이 미미 (v1)
4. **에이전트 이름만으로는 정확도 50%** — 설명 없이 이름만 나열하면 때려맞추기 수준 (v1)
5. **익숙하지 않은 이름은 회피** — `db-query-helper`, `migration-planner` 선택률 0% (v1)
6. **적합/비적합 태그가 효과 있음** — v1에서 전멸한 D06이 detailed에서 성공 (v2)
7. **에이전트 .md의 description 프론트매터가 핵심** — CLAUDE.md 설명보다 영향력 큼 (v3)
8. **CLAUDE.md 설명 수준은 부차적** — name-only(75%)가 detailed(68%)보다 높음 (v3)

### 02 — 훅 기반 선택
9. **forced-eval 훅으로 82%** — description만으로 60%, 훅 주입으로 +22%p (v4, N=40)
10. **"사고 과정 강제" > "정답 알려주기"** — Haiku가 정답 추천해도 Claude가 무시. llm-eval 70% = simple과 동일 (v4)
11. **direct 유형은 100% 해결 가능** — forced-eval에서 명시적 요청은 완벽 처리 (v4)
12. **ambiguous 유형은 50% 천장** — 모호한 요청은 훅으로도 한계 (v4)

### 02 — Step 0 분류
13. **Step 0 요청 분류로 87%** — "분류 먼저, 평가 나중" 패턴 유효 (v5, N=40)
14. **키워드 편향은 사전 분류로 극복** — "느려"→perf-opt 편향을 "query-level" 분류로 해결 (v5)
15. **natural 94%까지 도달** — v1 50% → v5 94%. 자연어도 거의 완벽 처리 (v5)
